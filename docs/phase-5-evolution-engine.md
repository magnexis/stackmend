# STACKMEND Phase 5 Evolution Engine

## Core principle

Phase 5 should turn STACKMEND into a continuously training software-learning organism.

It should not improve through prompts.

It should evolve through a closed empirical loop:

```text
Study
-> Attempt
-> Execute
-> Test
-> Compare
-> Extract technique
-> Retest
-> Promote technique
-> Apply to real projects
-> Measure outcome
```

Its intelligence should come from:

- structural program analysis
- deterministic transformations
- search algorithms
- statistical learning
- repeated verification
- measured outcomes

## Core subsystems

### Mend Academy

Internal training institution for:

- algorithms
- debugging
- refactoring
- dependency repair
- API integration
- database migrations
- concurrency
- security
- performance
- testing
- build systems
- configuration
- framework upgrades
- legacy modernization
- fault localization
- incomplete project repair

### Curriculum Director

Chooses what to study next from:

- weakness severity
- real-world relevance
- failure frequency
- uncertainty
- knowledge freshness

### Source Scholar

Studies approved sources through permission-aware ingestion only.

Preferred acquisition channels:

- official APIs
- official CLIs
- public repositories
- permissively licensed datasets
- user-exported exercises
- documentation feeds
- package registries
- release notes
- standards documents

### Evolution Arena

Sandboxed runtime for:

- coding examinations
- mutation tournaments
- transfer tests
- adversarial evaluations
- performance checks
- property checks

### Deterministic synthesis

Preferred solution mechanisms:

- strategy libraries
- templates
- enumerative search
- constraint solving
- repair search
- case-based reasoning
- bounded genetic programming
- search-based candidate ranking

## Examination classes

- technique tests
- repair tests
- mutation tests
- regression tests
- transfer tests
- adversarial tests

The goal is not to memorize answers.

The goal is to extract reusable, verified techniques.

## Technique extraction

After each attempt, STACKMEND should extract reusable `Technique Capsules` containing:

- problem family
- preconditions
- procedure
- failure modes
- verification requirements
- confidence

## Promotion

Candidate techniques should move through:

```text
Observed
-> Attempted
-> Locally successful
-> Variation tested
-> Transfer tested
-> Regression tested
-> Repository validated
-> Promoted
```

Promoted high-confidence techniques may become `Code Reflexes`.

## Source and contamination rules

Phase 5 must enforce:

- attempt-before-reference
- permission-aware source access
- no blind scraping
- no private account impersonation
- contamination tracking
- hidden final evaluation where possible

## Final identity

At this stage, STACKMEND becomes:

- a non-prompt-driven software-learning bot
- an evidence-backed repair and synthesis system
- a continuously examined technique engine
- a verified library of coding and repair reflexes
